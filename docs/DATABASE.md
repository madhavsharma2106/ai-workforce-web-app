# Database

Supabase (Postgres) backs both auth and persistence. Auth is wired via `@supabase/ssr` (`src/lib/supabase/client.ts`, `src/lib/supabase/server.ts`); this doc covers the persistence side — schema and how to change it.

## Env vars

- `NEXT_PUBLIC_SUPABASE_URL`
- `NEXT_PUBLIC_SUPABASE_ANON_KEY`
- `SUPABASE_SERVICE_ROLE_KEY` — server/job-only, never imported client-side (see `src/lib/supabase/admin.ts`). Used by background jobs (Inngest), which run with no user session and so bypass RLS entirely — job code must filter every query by `user_id`/`employee_id` itself. See [AGENTS.md](AGENTS.md).

Find all three in the Supabase project settings under API. See `.env.example`.

## CLI

The Supabase CLI is a devDependency, invoked via `npx supabase ...` — no global install needed.

One-time setup per machine:

```
npx supabase login                            # opens a browser to authorize the CLI
npx supabase link --project-ref <project-ref> # project-ref is in the dashboard URL/settings
```

## Migrations

Schema changes are SQL files under `supabase/migrations/`, applied to the linked remote project with:

```
npx supabase db push
```

To add a new migration, create a new timestamped file in `supabase/migrations/` (e.g. `date -u +%Y%m%d%H%M%S` for the prefix) and run `db push` again. Don't edit past migrations — add a new one.

Check current state with `npx supabase migration list` (compares local files against what's applied remotely).

| `leads`             | One qualified lead the Lead Sourcer's `save_lead` tool persisted during an `agent_run`, later drafted by the Sales Representative's `draft_outreach_email` tool once approved | `id` (PK), `user_id`, `employee_id`, `run_id` (→ `agent_runs`, no separate `days` table — one run _is_ one day's work), `company`, `website`, `fit`, `decision_maker`, `email`, `draft`, `sources`, `person_id`, `email_revealed`, `status` (`pending` \| `approved` \| `rejected` — Emma's qualification gate), `draft_status` (`pending` \| `approved` \| `rejected` — Oliver's separate draft-review gate), `draft_run_id` (→ `agent_runs`, the run that last wrote `draft`), `feedback_reason`, `research_snippet` (a literal detail from the site research, or an honest note that none was found), `industry`, `employee_count`, `location`, `founded_year`, `company_linkedin_url`, `contact_linkedin_url`, `contact_seniority`, `contact_departments` (`text[]`) — all copied through from the same Apollo response `search_leads` already fetched, all nullable since Apollo doesn't always have them, `created_at`, `updated_at`. `draft` is empty until Oliver drafts it — `save_lead` no longer writes it. `employee_id` stays Emma's id even after Oliver drafts; Oliver's queue (`getLeadsAwaitingOutreach`) is defined by `status = 'approved'`, not row ownership. Candidates Emma reviewed but didn't qualify aren't rows here — see the `note_passed_candidates` tool in [AGENTS.md](AGENTS.md), captured via `agent_run_steps` instead. |

`business_profiles` now has real writers/readers: Account Manager onboarding and her `/employee/:id` dashboard's "Update profile" flow (`/api/employees/[id]/complete-onboarding` and `/api/employees/[id]/business-profile`) both write it via `synthesizeBusinessProfile` in [src/lib/businessProfile.ts](../src/lib/businessProfile.ts); every role's system prompt includes `profile_md` (`buildSystemPrompt` in [src/lib/agents/systemPrompt.ts](../src/lib/agents/systemPrompt.ts)), which is how the Lead Sourcer derives the keyword phrase it passes to its `search_leads` tool call — see [src/lib/agents/tools/searchLeadsTool.ts](../src/lib/agents/tools/searchLeadsTool.ts).

`employees.instructions_md` follows the same shape but scoped to one employee rather than the whole business: the Lead Sourcer's and Sales Representative's onboarding (`complete-onboarding`) synthesizes it from that employee's own onboarding transcript via `synthesizeEmployeeInstructions` in [src/lib/employeeInstructions.ts](../src/lib/employeeInstructions.ts), it's editable afterward from that employee's own page (`/api/employees/[id]/instructions`), and `buildSystemPrompt` includes it alongside `profile_md` so it actually shapes every future run/turn, not just the onboarding conversation that produced it.

`agent_runs` and `agent_run_steps` are on the Supabase Realtime publication, so the frontend could subscribe to a run's progress live — nothing in the app consumes Realtime yet, though; `leads` isn't on the publication either, and `LeadSourcerHome` currently polls `/api/employees/[id]/latest-run` while a run is in progress rather than subscribing. See [AGENTS.md](AGENTS.md) for how these tables get written to (the LangGraph delegation graph, run by Inngest jobs).

RLS is enabled on all tables; policies scope rows to `auth.uid() = user_id`, so each user only sees their own data. Background jobs (Inngest) don't have a user session and use a service-role client that bypasses RLS — see the `SUPABASE_SERVICE_ROLE_KEY` note above.
